# Order

In this guide we will take you through the steps of ordering a HANA Cloud instance and setting it up using the [Crossplane HANA provider](https://github.tools.sap/cloud-orchestration/crossplane-provider-hana).

:::note
You are an platform operator or administrator? Feel free to checkout the [OpenMCP project](https://openmcp-project.github.io/docs/) it's a project that makes it easier to enable more teams in your organization to use Crossplane by providing a managed control plane as a service.
:::

## 🚧 Prerequisites

- You have a running control plane.
- You have setup the [BTP provider](/docs/crossplane-provider-btp/docs/user-stories/setup/configure-provider-btp)
- You have setup a [managed subaccount](/docs/crossplane-provider-btp/docs/user-stories/account/subaccount)

## Service Entitlement
To create a `ServiceInstance` of HANA Cloud, you need to make sure that your subaccount has the service entitlement.

```yaml {8,9,14}
apiVersion: account.btp.sap.crossplane.io/v1alpha1
kind: Entitlement
metadata:
  name: hana-cloud-entitlement
spec:
  forProvider:
    serviceName: hana-cloud
    servicePlanName: hana # select the plan that fits your needs
    servicePlanUniqueIdentifier: hana-cloud-hana # required if your service plan name is not unique across regions like "hana-cloud-hana" or "hana-cloud-hana-sap_eu-de-1"
    enable: true
    subaccountRef:
      name: my-subaccount
  providerConfigRef:
    name: account-provider-config # Please reference your account providerConfig here!
```
:::note `servicePlanName`not unique?
Don't forget to set the `spec.forProvider.servicePlanUniqueIdentifier` field. Use `servicePlanUniqueIdentifier: hana-cloud-hana-sap_eu-de-1` for the SAP Europe Data Center or `servicePlanUniqueIdentifier: hana-cloud-hana` for other supported data centers.
:::

## Install Provider

In this chapter we manage our HANA Cloud service using the [Crossplane HANA provider](https://github.com/SAP/crossplane-provider-hana) - a fork of the [Crossplane SQL provider](https://github.com/crossplane-contrib/provider-sql) .

<Tabs>
  <TabItem value="managed" label="managed">
      ```yaml
      apiVersion: core.openmcp.cloud/v1alpha1
      kind: ManagedControlPlane
      metadata:
        name: <MCP_NAME>
        namespace: <PROJECT_WORKSPACES_NAMESPACE>
      spec:
        components:
          crossplane:
            version: :mcp_cv_crossplane:
            providers:
              - name: hana
                version: :mcp_cv_provider-hana:
      ...
      ```
  </TabItem>

## Configure ProviderConfig

In the next step, we authenticate our control plane to orchestrate our HANA Cloud service instance.

```yaml title="examples/provider/config.yaml"
apiVersion: v1
kind: Secret
metadata:
  namespace: default
  name: hana-provider-secret
type: Opaque
data:
  endpoint: abcdefgh-base64-encoded-xyz== # E.g. Base64-encode: my-hana-domain.prod-eu10.hanacloud.ondemand.com
  port: NDQz # E.g. Base64-encode: 443
  username: REJBRE1JTg== # E.g. Base64-encode: DBADMIN 
  password: Q2xvdWQtMTIzNDUh # Same password as set during service instance creation E.g. Base64-encode: Cloud-12345!
---
apiVersion: objects.hana.orchestrate.cloud.sap/v1alpha1
kind: ProviderConfig
metadata:
    name: hana-providerconfig
spec:
  credentials:
    source: Secret
    connectionSecretRef:
      namespace: default
      name: hana-provider-secret
```

Apply the provider configuration by running

```sh
kubectl apply -f examples/provider/config.yaml
```

:::danger Give control plane smart access rights only

In this initial setup, you are providing your control plane ultimate Admin access to your database instance.
This is not ideal, as there is **no need for your control plane to have system admin access**.

**Instead, you want to create a new `User` that only inherits rights it needs - nothing more!**

:::
